# Multi-Criteria Query: PostgreSQL Planning Complexity

Investigation of the PostgreSQL planning bottleneck on the multi-criteria KGQuery
and options to reduce planning time from ~655ms to sub-100ms.

**Date**: 2026-03-08
**Dataset**: Lead dataset (`sp_sql_lead_dataset`, 100 entities, 192,810 triples)
**Benchmark script**: `vitalgraph_sparql_sql/scripts/lead_multi_criteria_v2.py`
**Query**: MQL=true + MQLRating≥65 + CompanyStateCode='CA' (10 results)

---

## 1. Problem Statement

The multi-criteria KGQuery generates SPARQL with a 4-branch UNION for entity
subtypes. After the v2 pipeline compiles this to SQL, PostgreSQL spends **655ms
planning** vs **38ms executing**. Wall time via the REST endpoint is ~3.4s.

The SPARQL (generated by `KGQueryCriteriaBuilder.build_entity_query_sparql()`):

```sparql
SELECT DISTINCT ?entity WHERE {
    GRAPH <urn:sql_lead_dataset> {
        { ?entity vital-core:vitaltype haley:KGEntity . }
        UNION { ?entity vital-core:vitaltype haley:KGNewsEntity . }
        UNION { ?entity vital-core:vitaltype haley:KGProductEntity . }
        UNION { ?entity vital-core:vitaltype haley:KGWebEntity . }

        # Frame criteria: LeadStatusFrame → LeadStatusQualificationFrame
        #   slot: MQLv2 = true
        #   slot: MQLRating >= 65.0
        # Frame criteria: CompanyFrame → CompanyAddressFrame
        #   slot: CompanyStateCode = "CA"
    }
}
```

Each UNION branch duplicates the **entire** frame/slot/edge body (~20 triple
patterns, 6 edge rewrites). The v2 generator produces 4 nearly-identical SQL
UNION ALL branches, yielding **~71.8k chars of SQL** with ~16 variables and
~24 JOINs per branch.

### Measured Timing Breakdown

| Phase | Time |
|-------|------|
| Sidecar compile | ~15ms |
| v2 SQL generation | ~25ms |
| **PG planning** | **647.6ms** (708 buffer hits) |
| PG execution | **38.4ms** |
| REST overhead + serialization | ~2,700ms |
| **Wall time (REST)** | **~3,400ms** |

The PG planning time is the primary bottleneck at the SQL level. The large
REST overhead (~2.7s) is a separate issue (likely JSON serialization of entity
graphs in the response assembly layer).

## 2. Root Cause Analysis

### 2.1 UNION Branch Explosion

`KGQueryCriteriaBuilder` always emits a 4-branch UNION for entity types
(`kg_query_builder.py` lines 295-304):

```python
entity_type_clause = """
{
    ?entity vital-core:vitaltype haley:KGEntity .
} UNION {
    ?entity vital-core:vitaltype haley:KGNewsEntity .
} UNION {
    ?entity vital-core:vitaltype haley:KGProductEntity .
} UNION {
    ?entity vital-core:vitaltype haley:KGWebEntity .
}"""
```

In most spaces, only `KGEntity` exists. The other 3 branches match 0 rows but
PostgreSQL must still **plan** all 4 branches. Each branch contains the full
frame/slot/edge pattern (~24 JOINs after edge rewrite), so the planner
evaluates ~96 JOIN combinations total.

### 2.2 Why PG Planning is Expensive Here

- **SQL size**: 71.8k chars — the planner must parse and analyze the entire AST
- **UNION ALL**: PG plans each branch independently, then merges
- **JOIN count per branch**: ~24 tables — approaches `geqo_threshold` (default 12)
- **GEQO**: Branches with 12+ tables trigger the genetic query optimizer,
  which is stochastic and slower than exhaustive search
- **DISTINCT + ORDER BY**: The outer DISTINCT and ORDER BY add a Sort + Unique
  node that must plan over the 4-branch UNION

### 2.3 Comparison: Simple vs Multi-Criteria

| Query | UNION branches | JOINs/branch | SQL chars | PG plan | PG exec | Wall |
|-------|---------------|-------------|-----------|---------|---------|------|
| Find MQL leads (1 frame) | 4 | ~12 | ~30k | ~80ms | ~20ms | 180ms |
| Multi-criteria (2 frames) | 4 | ~24 | ~72k | **655ms** | 38ms | 3,400ms |

Planning time scales super-linearly with JOINs-per-branch because PG's planner
is O(n!) for exhaustive search and O(n²) for GEQO.

### 2.4 DB Round-Trips During v2 SQL Generation

The v2 `generate_sql()` pipeline makes **5 DB round-trips** before the actual
query executes. These happen between sidecar compile and SQL emission:

| Stage | Function | Purpose | Per-query? |
|-------|----------|---------|------------|
| 2 | `materialize_constants()` | Resolve URI/literal strings → UUIDs via `{space}_term` | Yes (constants vary per query) |
| 2a | `_load_quad_stats()` | Load pred_stats + quad_stats for join reordering | No (✅ cached globally per space) |
| 2a.1 | `ensure_edge_table()` | Check if edge table exists | Yes (could be cached) |
| 2a.2 | `ensure_frame_entity_table()` | Check if frame_entity table exists | Yes (could be cached) |
| 2b | `_load_datatype_cache()` | Load datatype_id → URI mapping | Yes (could be cached) |

**Key insight**: `materialize_constants()` already queries the term table for
**every** URI constant in the SPARQL, including the 4 entity type URIs
(`haley:KGEntity`, `haley:KGNewsEntity`, etc.). When a constant resolves to
**no UUID** (not in the term table), we already know that UNION branch is dead.
Currently the code logs a warning and continues — but this information could be
used to prune dead UNION branches from the IR tree before SQL emission, at
**zero additional DB cost**.

This is captured in Option F below.

## 3. Proposed Solutions

### Option A: UNION Branch Pruning at SPARQL Generation Layer

**Where**: `kg_query_builder.py` → `build_entity_query_sparql()`

**Approach**: Before building the SPARQL, probe the space's term/quad table to
determine which entity subtypes actually exist. Only emit UNION branches for
types that have ≥1 row.

```python
# Probe query (cheap — index lookup on term table)
SELECT DISTINCT t.term_text
FROM {space}_rdf_quad q
JOIN {space}_term t ON q.object_uuid = t.term_uuid
WHERE t.term_text IN (
    'http://vital.ai/ontology/haley-ai-kg#KGEntity',
    'http://vital.ai/ontology/haley-ai-kg#KGNewsEntity',
    'http://vital.ai/ontology/haley-ai-kg#KGProductEntity',
    'http://vital.ai/ontology/haley-ai-kg#KGWebEntity'
)
```

For the lead dataset this returns only `KGEntity`, reducing UNION from 4→1
branches. SQL shrinks from 72k→18k chars, JOINs from 96→24, and PG planning
from 655ms→~80ms.

**Pros**:
- Simple, surgical change in one file
- Probe query is < 1ms (indexed)
- No changes to v2 pipeline or SQL generation
- Works for all KGQuery patterns, not just multi-criteria

**Cons**:
- Requires async DB access in the SPARQL builder (currently synchronous)
- Adds a DB round-trip per query (amortizable with caching)
- Only helps when subtypes are absent — if all 4 exist, no improvement

**Estimated impact**: 655ms → ~80ms planning (8x), 3.4s → ~0.8s wall (4x)

### Option B: UNION Branch Pruning at SQL Level (v2 Generator)

**Where**: `vitalgraph/db/sparql_sql/generator.py` — after SQL generation

**Approach**: After generating the full SQL with 4 UNION ALL branches, analyze
each branch to detect constant predicates (`object_uuid = <known-absent-uuid>`)
and eliminate branches that provably match 0 rows.

This is essentially the same probe as Option A but applied later in the pipeline
— after the SPARQL has been compiled and the SQL generated. The generator already
has `conn` available for DB access.

**Pros**:
- No changes to SPARQL builder
- Can be implemented as a post-processing step on the SQL string or IR
- Generator already has DB connection and space context

**Cons**:
- Harder to implement (must parse/manipulate SQL or the IR tree)
- SQL is already generated — waste of work for pruned branches
- Still pays the sidecar compile cost for all 4 branches

**Estimated impact**: Same as Option A for planning, but slightly worse for
total latency (sidecar still compiles all 4 branches).

### Option C: Cached Entity Type Set Per Space

**Where**: `KGQueryCriteriaBuilder` or the endpoint layer

**Approach**: Maintain a cached set of entity types per space_id. On first query,
probe the DB and cache. On subsequent queries, use the cached set to skip absent
types. Invalidate on write operations (insert/delete).

```python
# Cache structure (global, per space_id)
_entity_type_cache: Dict[str, Set[str]] = {}

def get_entity_types(space_id: str, conn) -> Set[str]:
    if space_id not in _entity_type_cache:
        _entity_type_cache[space_id] = _probe_entity_types(space_id, conn)
    return _entity_type_cache[space_id]
```

**Pros**:
- Zero DB overhead on warm queries (probe only on first query or after invalidation)
- Simple invalidation — clear on any write to the space
- Combines well with Option A

**Cons**:
- Cache management complexity (invalidation, TTL)
- Stale cache could produce incorrect results if new entity types are added
  between cache load and query (window is small, result is incomplete not wrong)

**Estimated impact**: Same as Option A, minus the per-query probe cost.

### Option D: Replace UNION with VALUES + JOIN

**Where**: `kg_query_builder.py`

**Approach**: Instead of 4 UNION branches, use a single BGP with a VALUES clause:

```sparql
SELECT DISTINCT ?entity WHERE {
    GRAPH <urn:sql_lead_dataset> {
        VALUES ?entityType {
            haley:KGEntity haley:KGNewsEntity
            haley:KGProductEntity haley:KGWebEntity
        }
        ?entity vital-core:vitaltype ?entityType .
        # ... frame/slot/edge patterns (single copy) ...
    }
}
```

This produces **one** SQL branch with a VALUES-based filter instead of 4 UNION
branches. PG plans a single query with ~24 JOINs (below GEQO threshold) and
filters on the entity type at the quad level.

**Pros**:
- Eliminates UNION entirely — SQL shrinks by ~75%
- Single plan path — PG plans once, not 4×
- No DB probe needed
- No cache needed
- Semantically equivalent to the UNION approach

**Cons**:
- Requires changes to how the Jena sidecar compiles VALUES (must verify support)
- The v2 generator must handle VALUES correctly (needs testing)
- PG may not optimize `IN (v1, v2, v3, v4)` as well as 4 separate scans
  if the type distribution is very skewed

**Estimated impact**: 655ms → ~100-150ms planning (4-6x), SQL 72k → ~18k chars.

### Option E: PostgreSQL `plan_cache_mode` / Prepared Statements

**Where**: Connection pool configuration or per-query hint

**Approach**: Use PostgreSQL prepared statements to cache the query plan. On
first execution, PG plans normally (~655ms). On subsequent executions with
different parameter values, PG reuses the cached plan (~0ms planning).

```python
# psycopg3 prepared statement
async with pool.acquire() as conn:
    cursor = conn.cursor()
    await cursor.execute(sql, prepare=True)  # first call: plan + execute
    await cursor.execute(sql, prepare=True)  # subsequent: execute only
```

**Pros**:
- Zero planning cost after first execution
- No changes to SPARQL builder or SQL generation
- Works for any query pattern, not just entity type UNIONs

**Cons**:
- v2 generates SQL with inline constants, not `$1` parameters — requires
  parameterization infrastructure (see `sparql_sql_v2_performance_plan.md` §5)
- Plan cache is per-connection — pool connections are shared, so plan reuse
  depends on connection stickiness
- Generic plans may be suboptimal vs custom plans for specific parameter values
- Significant implementation effort for the parameterization layer

**Estimated impact**: 655ms → 0ms planning on cache hit, but only for repeated
executions of the same query structure.

## 4. Recommendation

### Immediate (P1): Option A + C — Probe & Cache Entity Types

Combine the probe (Option A) with caching (Option C) for minimal implementation
effort and maximum impact:

1. Add `get_space_entity_types(space_id, conn)` utility that probes the term
   table for existing entity subtypes
2. Cache per space_id, invalidate on write operations
3. In `KGQueryCriteriaBuilder.build_entity_query_sparql()`, accept an optional
   `entity_types: Set[str]` parameter. If provided, only emit UNION branches
   for types in the set. If only one type exists, emit a plain triple pattern
   (no UNION at all).

**Estimated effort**: ~2 hours. Changes to 2 files:
- `kg_query_builder.py` — conditional UNION emission
- Endpoint or backend layer — probe + cache

**Expected result**: Multi-criteria query drops from 3.4s → ~0.8s wall time.

### Option F: Batched EXISTS Probing in the v2 Generator (Recommended)

**Where**: `vitalgraph/db/sparql_sql/generator.py` — between Stage 2
(materialize_constants) and Stage 3 (emit)

**Approach**: After the IR tree is built and constants are materialized, detect
UNION nodes whose branches differ only by a single constant (the entity type
URI). For each such constant, run a batched `EXISTS` probe against the quad
table to check whether any row matches. Prune branches where the probe returns
0 rows.

The probe can be batched into a **single SQL statement** with one `EXISTS` per
entity type:

```sql
SELECT
    EXISTS(SELECT 1 FROM {space}_rdf_quad
           WHERE predicate_uuid = '{vitaltype_uuid}'
             AND object_uuid = '{KGEntity_uuid}'
             AND context_uuid = '{graph_uuid}'
           LIMIT 1) AS has_kgentity,
    EXISTS(SELECT 1 FROM {space}_rdf_quad
           WHERE predicate_uuid = '{vitaltype_uuid}'
             AND object_uuid = '{KGNewsEntity_uuid}'
             AND context_uuid = '{graph_uuid}'
           LIMIT 1) AS has_kgnewsentity,
    EXISTS(SELECT 1 FROM {space}_rdf_quad
           WHERE predicate_uuid = '{vitaltype_uuid}'
             AND object_uuid = '{KGProductEntity_uuid}'
             AND context_uuid = '{graph_uuid}'
           LIMIT 1) AS has_kgproductentity,
    EXISTS(SELECT 1 FROM {space}_rdf_quad
           WHERE predicate_uuid = '{vitaltype_uuid}'
             AND object_uuid = '{KGWebEntity_uuid}'
             AND context_uuid = '{graph_uuid}'
           LIMIT 1) AS has_kgwebentity
```

This is a single DB round-trip. Each `EXISTS ... LIMIT 1` short-circuits on the
first match — cost is essentially one index probe per type (~0.01ms each on the
`idx_quad_po` index). The UUIDs for `vitaltype`, `KGEntity`, etc. are already
resolved by `materialize_constants()` at this point, so we can use them directly.

Alternatively, `materialize_constants()` already tells us which constants
resolved to a UUID and which didn't. If `KGNewsEntity` has no entry in the
term table, then it certainly has no quads — the branch is provably dead
**without any additional query**. The EXISTS probe is only needed for types
that *are* in the term table but might have 0 quads with `vitaltype` predicate.

**Implementation sketch**:

```python
# In generate_sql(), after materialize_constants():

# 1. Check which entity type constants resolved to UUIDs
unresolved = {text for (text, _), col in aliases.constants.items()
              if col not in aliases.resolved_constants}
# Any UNION branch using an unresolved constant → prune immediately

# 2. For resolved constants, optionally run batched EXISTS probe
#    (only if the IR has a UNION node with entity-type-only branches)

# 3. Walk the IR tree, remove dead UNION children
plan = prune_dead_union_branches(plan, dead_types)
```

**Pros**:
- **Zero or one additional DB round-trip** (the `materialize_constants` check
  is free; the EXISTS probe is one batched query if needed)
- Works at the IR level — cleaner than SQL string manipulation
- No changes to SPARQL builder (stays synchronous)
- Generator already has `conn`, space_id, and resolved UUIDs
- Applies to all UNION patterns, not just entity types

**Cons**:
- Requires IR tree manipulation (identify + remove UNION children)
- Must correctly handle the case where all branches are pruned (should not happen
  for entity types, but needs a guard)

**Estimated impact**: Same as Option A (655ms → ~80ms planning), with near-zero
additional latency. The unresolved-constant check alone would handle the lead
dataset case (3 of 4 types are absent from the term table entirely).

## 4. Implementation Results: Option F (UNION Branch Pruning)

### 4.1 Implementation

Option F was implemented in `vitalgraph/db/sparql_sql/prune_union.py` and wired
into `generator.py` after `materialize_constants()` (Stage 2 post).

The pruning works correctly:
- **3 dead branches pruned** (KGNewsEntity, KGProductEntity, KGWebEntity)
- **0 UNION ALL** in generated SQL (down from 3)
- SQL size: 72k → 60k chars

### 4.2 Critical Finding: The UNION Was Not the Bottleneck

The Jena sidecar produces a **factored** algebra where the 4-branch UNION only
contains the tiny entity-type BGPs (2 tables each). The massive frame/slot/edge
pattern body is a **separate** BGP joined via `?entity`:

```
slice
  distinct
    project
      order
        filter
          join
            union               ← 4 tiny BGPs (2 tables each = 8 tables)
              union
                union
                  bgp (KGEntity)
                  bgp (KGNewsEntity)
                bgp (KGProductEntity)
              bgp (KGWebEntity)
            bgp (47 tables)     ← ALL frame/slot/edge patterns (the real cost)
```

The sidecar **does not** duplicate the 47-table body into each UNION branch.
It factors it out into a JOIN. So pruning 3 tiny branches (6 tables) from a
~55-table query barely moves the needle:

| Metric | Before pruning | After pruning | Change |
|--------|---------------|---------------|--------|
| UNION ALL branches | 3 | **0** | ✅ eliminated |
| SQL chars | ~72k | **~60k** | -17% |
| Total tables | ~55 | ~49 | -11% |
| **PG Planning Time** | **648ms** | **723ms** | **❌ no improvement** |
| PG Execution Time | 38ms | 37ms | (unchanged) |
| Rows | 10 | 10 | ✅ correct |

### 4.3 Root Cause Update

The planning bottleneck is the **47-table frame/slot/edge BGP**, not the UNION.
After edge MV rewrites this becomes ~30 tables, which is still 2.5× the GEQO
threshold (12). The PG planner spends ~650ms planning the join order for this
single massive BGP regardless of whether the UNION exists.

The UNION pruning is still a **correct and useful** optimization:
- It eliminates unnecessary SQL generation and PG parsing overhead
- It reduces SQL size by 12k chars
- For simpler queries (single-frame criteria), the UNION was a larger fraction
  of the total and pruning would help more
- It prevents regressions if more entity subtypes are added

But it **does not solve** the multi-criteria planning bottleneck.

## 5. Solution: text_needed_vars Optimization

### 5.1 Root Cause: Unnecessary Term JOINs

After the edge MV rewrite, the 47-table BGP becomes ~30 tables. Of those,
**18 were term table JOINs** for internal-only variables — variables used
solely for UUID-level joining between quad patterns but never projected,
filtered, or referenced by any expression.

Each term JOIN adds a table to the PG planner's search space. Removing
18 unnecessary JOINs drops the table count from ~43 to ~29, which crosses
below the threshold where PG's planner becomes efficient.

### 5.2 Why text_needed_vars Was Disabled

The `compute_text_needed_vars()` function in `var_scope.py` was disabled
(hardcoded to `None`) because the DISTINCT handler incorrectly marked ALL
descendant BGP vars as referenced:

```python
# BROKEN (original): defeats the optimization for any SELECT DISTINCT query
if kind in (KIND_DISTINCT, KIND_REDUCED, KIND_UNION, KIND_MINUS):
    _collect_all_bgp_vars(plan, refs)
```

Since virtually every KGQuery uses `SELECT DISTINCT`, this made every variable
"referenced" and no term JOINs could be skipped.

### 5.3 Fixes Applied (4 files)

**`var_scope.py`** — 4 fixes to `compute_text_needed_vars`:

1. **DISTINCT handler**: Use child's visible scope (the PROJECT output columns)
   instead of all descendant BGP vars. DISTINCT only deduplicates over the
   columns it receives from its child, not internal join variables.

2. **`_has_project` guard**: Walk only the root modifier chain. A PROJECT inside
   a JOIN child is a subquery boundary, not the outer query's PROJECT. Prevents
   `SELECT *` queries with inner subqueries from incorrectly skipping the
   "resolve all vars" fallback.

3. **`COUNT(*)` aggregates**: When an aggregate expression is `None` (i.e.,
   `COUNT(*)` or `COUNT(DISTINCT *)`), add all child scope variables to the
   referenced set. The `*` references all binding variables.

4. **`SELECT *` fallback**: When no PROJECT exists in the root modifier chain,
   return all BGP vars (every variable is projected to the output).

**`generator.py`** — Re-enabled `compute_text_needed_vars()`:

```python
# Stage 2c: Compute text-needed vars (skip term JOINs for internal-only vars)
from .var_scope import compute_text_needed_vars
text_needed = compute_text_needed_vars(plan)
```

**`emit_bgp.py`** — Fixed UUID passthrough for null companions:

`null_companions()` produces typed nulls (`NULL::uuid AS v0__uuid`), but the
override check used exact string match (`NULL AS v0__uuid`). Changed to
`.endswith(f" AS {sn}__uuid")` so the UUID from the inner query is correctly
passed through even when the term JOIN is skipped.

**`emit_join.py`** — 2 fixes:

1. **ON clause IS NULL check**: Use `__uuid` column instead of the base/text
   column. The base column is NULL when text_needed_vars skips the term JOIN,
   even though the variable IS bound. Using UUID correctly detects unbound
   variables for SPARQL compatible-mapping semantics.

2. **Propagate `from_triple`**: Register output variables with
   `from_triple=True` so downstream JOINs (chained OPTIONALs) know the
   variable has a UUID column available.

## 6. UNION Branch Pruning (Option F)

Implemented in `prune_union.py` as a complementary optimization. Prunes UNION
branches that reference constants absent from the term table (e.g., entity
subtypes not present in the dataset).

For the lead dataset: prunes 3 of 4 branches (KGNewsEntity, KGProductEntity,
KGWebEntity are absent). This reduces SQL size by ~12k chars and eliminates
unnecessary PG parsing, but was **not the planning bottleneck** — the term JOIN
elimination was.

## 7. Remaining Opportunities

1. **Prepared Statements**: Cache PG plans for repeated query structures.
   Would reduce planning from ~89ms to ~0ms on subsequent executions.

2. **LATERAL JOIN Rewrite**: For multi-criteria queries, force execution order
   per criteria path to reduce planner search space further.

3. **VALUES Replacement (Option D)**: Eliminate UNION at the SPARQL level.
   Belt-and-suspenders with the pruning approach.
