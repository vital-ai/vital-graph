- **KGFrame** (105) — frames attached to entities (e.g., CEO, CTO, address)
- **KGSlot** (various) — typed data slots on frames (text, datetime, integer, URI, entity)
- **Edge_hasKGSlot** (250) — edges connecting frames to slots
- **Edge_hasKGFrame** (30) — edges connecting entities to frames
- **Edge_hasEntityKGFrame** (60) — edges connecting entities to entity frames

### SPARQL vs SQL Result Differences

When comparing Fuseki SPARQL results to SQL results, the **literal representations
will differ** due to the datatype mismatch:

| What | Fuseki SPARQL Result | SQL Result (from `term_text`) |
|---|---|---|
| Integer | `{"type": "literal", "value": "100000", "datatype": "xsd:integer"}` | `"100000"` (plain string) |
| Boolean | `{"type": "literal", "value": "false", "datatype": "xsd:boolean"}` | `"false"` (plain string) |
| DateTime | `{"type": "literal", "value": "2026-01-21T07:53:01+00:00", "datatype": "xsd:dateTime"}` | `"2026-01-21T07:53:01+00:00"` (plain string) |
| Decimal | `{"type": "literal", "value": "19616.07", "datatype": "xsd:decimal"}` | `"19616.07"` (plain string) |
| Text | `{"type": "literal", "value": "Doe (TEST)"}` | `"Doe (TEST)"` (plain string) |

The comparison test normalizer must:
1. **Strip datatype annotations** when comparing — match on `value` only
2. **Normalize numeric formatting** — Fuseki may return `"0.0"` vs `"0"`
3. **Normalize datetime formatting** — timezone offset differences (e.g., `+00:00` vs `Z`)
4. **Ignore datatype metadata** — SQL results won't have `datatype` field
5. **Sort result rows** — row order may differ between Fuseki and SQL

## Key Observations

1. **No namespace/datatype tables** — these tables don't exist in this database.
   The plan assumed `{prefix}__{space_id}__namespace` etc. but they were never created.
2. **Fuseki has XSD datatypes; PostgreSQL does not** — Fuseki stores typed literals
   with XSD datatype annotations, but the PostgreSQL term table stores everything as
   plain strings (`datatype_id` is always NULL). This is a critical difference for
   SPARQL-to-SQL translation.

   | Predicate | Fuseki Literal | PostgreSQL `term_text` |
   |---|---|---|
   | `hasTextSlotValue` | plain literal (no type) | `"Doe (TEST)"` |
   | `hasIntegerSlotValue` | `"100000"^^xsd:integer` | `"100000"` |
   | `hasDateTimeSlotValue` | `"2026-01-21T07:53:01+00:00"^^xsd:dateTime` | `"2026-01-21T07:53:01+00:00"` |
   | `hasBooleanSlotValue` | `"false"^^xsd:boolean` | `"false"` |
   | `hasDoubleSlotValue` | `"0.0"^^xsd:decimal` | `"0.0"` |
   | `hasCurrencySlotValue` | `"19616.07..."^^xsd:decimal` | `"19616.07..."` |
   | `hasChoiceSlotValue` | plain literal | plain string |
   | `hasJsonSlotValue` | plain literal | plain string |

   **Implications for SQL generation**:
   - SPARQL `FILTER(?value > 50000)` on an integer slot requires
     `CAST(term_text AS BIGINT)` in SQL
   - SPARQL `FILTER(?date > "2026-01-01"^^xsd:dateTime)` requires
     `CAST(term_text AS TIMESTAMP)` in SQL
   - SPARQL `FILTER(?flag = true)` on a boolean slot requires
     `term_text = 'true'` in SQL (string comparison)
   - The predicate name gives the type hint needed for correct casting
   - For Fuseki comparison tests, results may differ in literal representation
     (e.g., Fuseki returns typed `"100000"^^xsd:integer`, SQL returns `"100000"`)
3. **No language tags** — `lang` column is always NULL.
4. **Only U and L term types** — no `G` (Graph) or `B` (BNode) terms in the data.
   Graph URIs are stored as regular URI terms.
5. **Single named graph per space** — most spaces use one context URI.
6. **All data uses Vital AI ontology** — predicates from `vital-core#`, `haley-ai-kg#`, `vital-aimp#`.
7. **Table naming** uses single underscore (`space_id_term`), not double underscore.
8. **Single dataset partition** — all terms in `dataset = 'primary'`.
